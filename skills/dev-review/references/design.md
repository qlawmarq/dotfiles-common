# 設計レビューガイド

## このガイドの対象

設計ドキュメント、実装プラン、API 設計、データモデル設計など、
ソフトウェアの構造的な設計判断を評価する際に使用する。

## チェックリスト

### 要件との整合性

- [ ] 設計が機能要件を全て満たしているか
- [ ] 非機能要件（性能、可用性、スケーラビリティ）が考慮されているか
- [ ] 制約条件が明示され、設計に反映されているか
- [ ] ユースケースやユーザーストーリーが設計でカバーされているか

### コンポーネント設計

- [ ] 各コンポーネントの責務が明確で単一責任原則（SRP）に従っているか
- [ ] コンポーネント間のインターフェースが明確に定義されているか
- [ ] コンポーネント間の結合度が適切に低いか（疎結合）
- [ ] 各コンポーネントの凝集度が高いか
- [ ] 依存関係が一方向か（循環依存がないか）

### インターフェース設計

- [ ] API のエンドポイント設計が一貫性を持っているか
- [ ] 入出力のデータ型が明確に定義されているか
- [ ] エラーレスポンスの形式が統一されているか
- [ ] バージョニング戦略が考慮されているか（必要な場合）
- [ ] 後方互換性が考慮されているか

### データモデル設計

- [ ] データの正規化が適切なレベルか
- [ ] データのライフサイクル（作成・更新・削除）が考慮されているか
- [ ] データの整合性制約が定義されているか
- [ ] インデックス戦略が考慮されているか
- [ ] マイグレーション戦略が明確か

### 拡張性・保守性

- [ ] 将来の要件変更に対応しやすい設計か
- [ ] 開放閉鎖原則（OCP）に従い、拡張に対して開かれているか
- [ ] 設計の複雑度が要件に見合っているか（YAGNI 原則）
- [ ] 設定の外部化が適切に行われているか

### エラーハンドリング設計

- [ ] 想定されるエラーケースが列挙されているか
- [ ] エラー伝播の戦略が明確か（例外、Result型、エラーコード）
- [ ] リトライ戦略が定義されているか（外部サービス連携時）
- [ ] フォールバック機構が考慮されているか

## レビュー手順

1. **要件の確認**: 設計が解決しようとしている問題と要件を理解する
2. **既存アーキテクチャの調査**: プロジェクトの既存パターンやアーキテクチャを調査し、設計がそれに沿っているか確認する
3. **公式ドキュメントの参照**: 使用するライブラリやフレームワークの公式ドキュメントを参照し、推奨パターンに沿っているか確認する
4. **トレードオフの評価**: 設計の各判断について、選択されなかった代替案とのトレードオフを評価する
5. **チェックリストの適用**: 上記チェックリストを順番に確認する
6. **改善提案の作成**: 問題点に対して、根拠と共に具体的な改善案を提示する

## よくある問題パターン

### 過度な抽象化 (Over-Engineering)

- **症状**: 現在の要件に対して過剰に複雑な設計
- **根拠**: YAGNI (You Aren't Gonna Need It) 原則 — 現時点で必要ない機能のために複雑さを導入しない
- **対処**: 現在の要件を満たす最もシンプルな設計を提案する

### God Object / God Module

- **症状**: 1つのコンポーネントに過剰な責務が集中
- **根拠**: 単一責任原則 (SRP) — 変更理由が複数あるコンポーネントは分割すべき
- **対処**: 責務ごとにコンポーネントを分割する設計を提案する

### 不適切な依存方向

- **症状**: 上位レイヤーが下位レイヤーの実装詳細に依存
- **根拠**: 依存性逆転原則 (DIP) — 抽象に依存し、具象に依存しない
- **対処**: インターフェースを導入し依存方向を修正する設計を提案する

### 暗黙の結合

- **症状**: コンポーネント間の依存がコード上で明示されず、実行時に初めて判明する
- **根拠**: 明示的な依存は保守性を向上させる
- **対処**: 依存性注入やイベント駆動パターンで明示的な結合に変更する設計を提案する

### 不十分なエラー設計

- **症状**: 正常系のみが設計され、異常系が考慮されていない
- **根拠**: 堅牢なシステムは障害モードを事前に設計する
- **対処**: 障害モード分析を行い、エラーハンドリング戦略を提案する
