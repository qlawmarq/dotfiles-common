# 実装レビューガイド

## このガイドの対象

実装済みのコード（プルリクエスト、コミット、ファイル変更）をレビューし、
コード品質、正確性、可読性を評価する際に使用する。

## チェックリスト

### 正確性・ロジック

- [ ] コードが意図した動作を正しく実装しているか
- [ ] エッジケースが適切に処理されているか（null/undefined、空配列、境界値）
- [ ] 条件分岐の網羅性（全てのケースがカバーされているか）
- [ ] ループの終了条件が正しいか（無限ループ、off-by-one エラー）
- [ ] 型の安全性が確保されているか（型変換、型チェック）
- [ ] 並行処理がある場合、競合状態やデッドロックが発生しないか

### エラーハンドリング

- [ ] 外部入力のバリデーションが適切か
- [ ] 例外/エラーが適切にキャッチされ処理されているか
- [ ] エラーメッセージがデバッグに十分な情報を含んでいるか
- [ ] エラーが握り潰されていないか（空の catch ブロック等）
- [ ] リソースの解放が保証されているか（finally、defer、using 等）

### 可読性・命名

- [ ] 変数名・関数名が意図を正確に伝えているか
- [ ] 関数が十分に短く、単一の責務を持っているか
- [ ] ネストが深すぎないか（3段階以上のネストは要検討）
- [ ] コメントが「なぜ」を説明しているか（「何を」は不要）
- [ ] マジックナンバーが定数として定義されているか

### コードの一貫性

- [ ] 既存のコードベースのパターンや規約に従っているか
- [ ] 命名規則がプロジェクト全体で統一されているか
- [ ] 同様の処理が既存コードと一貫した方法で実装されているか
- [ ] インポート順序やファイル構成が慣習に沿っているか

### 変更の適切さ

- [ ] 変更範囲が最小限か（関係ない変更が混入していないか）
- [ ] リファクタリングと機能変更が分離されているか
- [ ] 不要なコード（デバッグ用コード、コメントアウト）が残っていないか
- [ ] TODO コメントがある場合、対応チケットが紐づいているか

## レビュー手順

1. **変更の概要把握**: `git diff` や PR の説明から変更の目的と範囲を理解する
2. **テストを先に読む**: テストコードを先に読み、期待される動作を理解する（Google のレビュープラクティスに基づく）
3. **主要な変更の確認**: 主要なロジック変更をチェックリストに基づいてレビューする
4. **既存パターンとの整合性**: プロジェクト内の類似コードを検索し、パターンの一貫性を確認する
5. **公式ドキュメント照合**: 使用しているライブラリ・フレームワークの API が公式ドキュメントの推奨に沿っているか確認する
6. **エッジケース検証**: 境界値、null/空値、大量データなどのエッジケースを検証する

## よくある問題パターン

### 空の catch / 握り潰されたエラー

- **症状**: `catch (e) {}` や `catch (e) { console.log(e) }` のみで復帰処理がない
- **根拠**: エラーの握り潰しは障害の原因特定を困難にする
- **対処**: エラーの性質に応じて、リスロー、ロギング+リカバリ、ユーザー通知のいずれかを提案する

### 防御的プログラミングの不足

- **症状**: 外部からの入力を検証せずに処理に使用
- **根拠**: 信頼境界の外側からのデータは常に検証が必要
- **対処**: 入力バリデーション層の追加を提案する

### 密結合な実装

- **症状**: テストでモックが困難、あるいは変更の影響範囲が広い
- **根拠**: 疎結合な設計はテスタビリティと変更容易性を向上させる
- **対処**: 依存性注入やインターフェース抽出を提案する

### コードの重複

- **症状**: 同じロジックが複数箇所に存在
- **根拠**: DRY (Don't Repeat Yourself) 原則 — ただし、2箇所程度の重複で無理に抽象化する必要はない（Rule of Three）
- **対処**: 3箇所以上の重複がある場合に共通化を提案する

### 過度に複雑な条件式

- **症状**: 3つ以上の条件が AND/OR で組み合わされた式
- **根拠**: 複雑な条件式はバグの温床であり、読み手の認知負荷を高める
- **対処**: 早期リターン、ガード句、意味のある名前の変数への分割を提案する

## git diff の活用

実装レビューでは `git` コマンドを活用して変更箇所を特定する:

```bash
# ステージされた変更の確認
git diff --staged

# 特定ブランチとの差分
git diff main...HEAD

# 変更されたファイル一覧
git diff --name-only main...HEAD

# 特定ファイルの変更履歴
git log --oneline -10 -- <file>
```
