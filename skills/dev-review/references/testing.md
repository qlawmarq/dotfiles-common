# テストレビューガイド

## このガイドの対象

テストコードの品質、カバレッジ、保守性を評価する際に使用する。
ユニットテスト、結合テスト、E2E テストなど全種類のテストが対象。
実装レビューにおいてテストの存在・妥当性を確認する場合にも使用する。

## チェックリスト

### テストの存在と網羅性

- [ ] 変更されたコードに対応するテストが存在するか
- [ ] 新機能に対するテストが追加されているか
- [ ] バグ修正に対する回帰テストが追加されているか
- [ ] 正常系だけでなく異常系のテストがあるか
- [ ] 境界値のテストがあるか（0, 1, 最大値, null, 空文字, 空配列）
- [ ] エラーパスのテストがあるか

### テストの品質

- [ ] テスト名が何をテストしているか明確に伝えているか
- [ ] 各テストが1つのことだけをテストしているか（単一アサーション原則）
- [ ] テストが実装の詳細ではなく振る舞いをテストしているか
- [ ] テストが独立しており、他のテストの結果に依存していないか
- [ ] テストの Arrange-Act-Assert (AAA) パターンが明確か
- [ ] テストが壊れにくいか（実装の軽微な変更でテストが壊れないか）

### テストの信頼性

- [ ] テストが決定的（deterministic）か（時刻、乱数、外部サービスに依存していないか）
- [ ] フレーキー（不安定）テストの原因がないか（タイミング依存、並行アクセス）
- [ ] 外部依存がモック / スタブされているか（データベース、API、ファイルシステム）
- [ ] テストデータが適切に設定・クリーンアップされているか

### テストの保守性

- [ ] テストヘルパーやファクトリがコード重複を削減しているか
- [ ] テストフィクスチャが再利用可能か
- [ ] テストが読みやすく、意図が明確か（テスト自体がドキュメントとして機能するか）
- [ ] テストファイルの構造がプロジェクトの規約に沿っているか

### テストの種類と戦略

- [ ] テストピラミッドに沿った適切な種類のテストか（ユニット > 結合 > E2E）
- [ ] 統合テストが必要な箇所で過度にモックしていないか
- [ ] E2E テストが真に必要な箇所にのみ使用されているか
- [ ] スナップショットテストが安易に使用されていないか（意図的な変更で壊れやすい）

## レビュー手順

1. **テストの目的理解**: テスト名と説明から、何をテストしようとしているかを理解する
2. **テスト対象コードの確認**: テスト対象の実装コードを読み、テストが適切にカバーしているか確認する
3. **エッジケースの検討**: テスト対象のロジックに対して、漏れているエッジケースがないか検討する
4. **テストの実行**: 可能であればテストを実行し、全て通過するか確認する
5. **フレームワーク規約の確認**: テストフレームワークの公式ドキュメントを参照し、推奨パターンに沿っているか確認する

## よくある問題パターン

### テストが実装の詳細に依存

- **症状**: 内部メソッドの呼び出し回数やプライベート状態を検証している
- **根拠**: Google Testing Blog — テストは振る舞い（公開 API の入出力）を検証すべき。実装詳細に依存するテストはリファクタリングを阻害する
- **対処**: 公開 API の入出力や副作用（DB への書き込み、イベントの発火）を検証するテストに書き換える

### 過度なモック

- **症状**: テスト対象以外の全てがモック化され、実際の統合動作が検証されていない
- **根拠**: モックが多すぎるテストは、モックの動作を検証しているだけになりがち
- **対処**: 重要な統合ポイントには結合テストを追加し、モックは外部サービスとの境界のみに限定する

### テスト名が不明確

- **症状**: `test1`, `testUpdate`, `it("works")` のような名前
- **根拠**: テスト名は失敗時に何が壊れたかを即座に伝える唯一の手がかり
- **対処**: `it("should return 404 when user does not exist")` のように、条件と期待結果を含む名前にする

### アサーションの欠落

- **症状**: テストが実行されるが何も検証していない（アサーションがない）
- **根拠**: アサーションのないテストは常に通過するため、回帰検出の役に立たない
- **対処**: 各テストに明確なアサーションを追加する

### フレーキーテスト

- **症状**: ランダムに成功/失敗するテスト
- **根拠**: フレーキーテストは CI の信頼性を損ない、本物の失敗が見逃される原因となる
- **対処**: 時刻依存は固定値やモッククロックに、並行処理はシリアライズまたは適切な同期に変更する
