# ベストプラクティス準拠レビューガイド

## このガイドの対象

コードが使用しているライブラリ、フレームワーク、言語の最新ベストプラクティスに
準拠しているかを評価する際に使用する。非推奨 API の検出、モダンパターンの適用、
公式推奨に沿った実装かどうかを確認する。

**重要**: このガイドのレビューでは、必ず Web 検索やドキュメント取得ツールを使用して最新の公式情報を確認すること。静的な知識だけでレビューしない。

## チェックリスト

### 非推奨 API・パターンの検出

- [ ] 使用している API やメソッドが非推奨（deprecated）になっていないか
- [ ] 非推奨パッケージへの依存がないか
- [ ] 古い構文やパターンが使用されていないか（例: `var` vs `let`/`const`、`componentWillMount` vs `useEffect`）
- [ ] マイグレーションガイドが公開されている非推奨機能を使用していないか
- [ ] コンパイラ/ランタイムの警告が発生する記述がないか

### 言語の最新機能の活用

- [ ] プロジェクトのターゲットバージョンで利用可能な最新の言語機能が適切に活用されているか
- [ ] 冗長な記述が新しい構文で簡潔にできないか
- [ ] 標準ライブラリで提供されている機能を外部ライブラリで代替していないか

### フレームワーク公式推奨パターン

- [ ] フレームワークの公式ドキュメントの推奨パターンに沿っているか
- [ ] フレームワーク提供のユーティリティやヘルパーが活用されているか
- [ ] フレームワークのアンチパターンに該当する実装がないか
- [ ] フレームワークのバージョンアップに伴う推奨変更が反映されているか

### 依存パッケージの健全性

- [ ] 依存パッケージが actively maintained か（最終更新日、GitHub stars、Issues の対応状況）
- [ ] 同じ機能を提供するより良い代替パッケージがないか
- [ ] 依存パッケージのバージョンが妥当か（最新のメジャーバージョンに追随しているか）
- [ ] ライセンスがプロジェクトの要件と互換性があるか

### エコシステムの慣習

- [ ] リンター/フォーマッターの設定がエコシステムの標準に沿っているか
- [ ] パッケージマネージャーの使い方が推奨に沿っているか
- [ ] ビルド設定やツールチェインが最新の推奨に沿っているか
- [ ] プロジェクト構成がフレームワークの推奨構成に沿っているか

## レビュー手順

1. **技術スタックの特定**: 使用している言語、フレームワーク、主要ライブラリを特定する
2. **公式ドキュメントの最新版を取得**: Web 検索または `resolve-library-id` → `query-docs` で最新の公式ドキュメントを取得する
3. **非推奨チェック**: 使用している API やパターンが非推奨になっていないか、公式のマイグレーションガイドや CHANGELOG を確認する
4. **推奨パターンとの比較**: 公式ドキュメントの推奨パターンと実装を比較する
5. **エコシステム動向の確認**: 必要に応じて、コミュニティでの議論や RFC を確認する
6. **改善提案の作成**: 非推奨箇所や推奨からの逸脱に対して、公式ドキュメントの URL を添えた具体的な改善案を提示する

## 情報収集の方法

### 必ず実施する情報収集

レビュー時に以下の手順で最新情報を収集すること:

1. **ライブラリのドキュメント取得**:
   - `resolve-library-id` で使用ライブラリを特定
   - `query-docs` で公式ドキュメントの該当セクションを取得

2. **Web 検索での最新情報確認**:
   - `[ライブラリ名] [バージョン] migration guide`
   - `[ライブラリ名] deprecated API`
   - `[ライブラリ名] best practices [年]`
   - `[ライブラリ名] changelog`

3. **公式リポジトリの確認**:
   - CHANGELOG.md / MIGRATION.md の確認
   - Breaking changes の確認
   - RFC や Discussions の確認

### 指摘には必ず公式ソースを添付

```
- **指摘**: `componentWillMount` は非推奨です
- **根拠**: React 公式ドキュメント (https://react.dev/reference/react/Component#componentwillmount)
- **改善案**: `useEffect` フックに置き換えてください
```

## よくある問題パターン

### 非推奨 API の使用

- **症状**: コンパイラ警告なしに非推奨 API を使用（警告が出ないバージョンで非推奨が宣言された場合）
- **根拠**: 非推奨 API は将来のバージョンで削除される可能性がある
- **対処**: 公式マイグレーションガイドに従って推奨 API に移行する。移行コストが高い場合はその旨を報告する

### レガシーパターンの踏襲

- **症状**: プロジェクト内の古いコードを参考に、既に非推奨のパターンで新しいコードを実装
- **根拠**: 新しいコードは最新のベストプラクティスで書くべき。古いコードの存在は現在の推奨とは限らない
- **対処**: 最新の公式ドキュメントに基づく実装を提案する。既存の古いコードも、今後の機会に更新する旨を Info レベルで提案する

### 車輪の再発明

- **症状**: 言語の標準ライブラリやフレームワークが提供する機能を自前で再実装
- **根拠**: 標準機能はテスト済み・最適化済み・ドキュメント完備であり、自前実装よりも信頼性が高い
- **対処**: 標準提供の機能を使用する実装に置き換える。カスタマイズが必要な場合は、標準機能を拡張する方法を提案する

### 古い依存パッケージの使用

- **症状**: メンテナンスが停止したパッケージ、または既に後継パッケージがあるパッケージを使用
- **根拠**: メンテナンスされていないパッケージはセキュリティリスクがあり、新しい環境での互換性問題を引き起こす
- **対処**: 後継パッケージまたは代替パッケージへの移行を提案する。移行の影響範囲と手順を添える

## 参考情報源

- 各言語・フレームワークの公式ドキュメント（レビュー時に必ず最新版を取得すること）
- 各ライブラリの CHANGELOG / マイグレーションガイド
- [Can I Use](https://caniuse.com/) — ブラウザ互換性の確認
- [Node.js Releases](https://nodejs.org/en/about/previous-releases) — Node.js バージョンのサポート状況
- [Python EOL](https://devguide.python.org/versions/) — Python バージョンのサポート状況
