# アーキテクチャ整合性レビューガイド

## このガイドの対象

コード変更がプロジェクトの既存アーキテクチャと整合しているかを評価する際に使用する。
レイヤー構成、依存方向、モジュール分割、技術的負債の観点から評価する。

## チェックリスト

### 既存アーキテクチャとの整合性

- [ ] 変更がプロジェクトの確立されたアーキテクチャパターンに従っているか
- [ ] ディレクトリ構成がプロジェクトの規約に沿っているか
- [ ] ファイルの配置が意味的に正しい場所か
- [ ] 既存の命名規約（ファイル名、クラス名、モジュール名）に従っているか
- [ ] 設定ファイルの管理方法が既存パターンと一貫しているか

### レイヤー構成と依存方向

- [ ] レイヤー間の依存が正しい方向か（上位→下位、外側→内側）
- [ ] ドメイン層がインフラ層に直接依存していないか
- [ ] プレゼンテーション層がデータアクセスを直接行っていないか
- [ ] 循環依存が発生していないか
- [ ] インターフェース/抽象を介した適切な依存性逆転が行われているか

### モジュール境界

- [ ] モジュールの責務が明確に分離されているか
- [ ] モジュール間の通信が定義されたインターフェースを通じて行われているか
- [ ] 内部実装詳細が適切にカプセル化されているか（外部に公開すべきでないものが公開されていないか）
- [ ] 共有コードが適切な場所（共通ライブラリ等）に配置されているか
- [ ] 新しいモジュールの導入が必要な場合、その粒度が適切か

### 一貫性

- [ ] 同じ種類の問題に対して既存と同じ解決パターンが使われているか
- [ ] エラーハンドリング戦略が既存パターンと一貫しているか
- [ ] ロギング方式が統一されているか
- [ ] 設定の取得方法が統一されているか
- [ ] テストの構成パターンが統一されているか

### 技術的負債

- [ ] 既存の技術的負債を悪化させていないか
- [ ] 一時的な回避策（ワークアラウンド）が恒久的な解決策として残されていないか
- [ ] TODO/FIXME/HACK コメントがある場合、対応計画が明確か
- [ ] 非推奨の内部パターンを新たに使用していないか

### スケーラビリティ

- [ ] 変更がシステムの水平/垂直スケーリングを阻害しないか
- [ ] ステートフルな依存が不必要に導入されていないか
- [ ] 設定のハードコードがスケーリング時に問題にならないか

## レビュー手順

1. **プロジェクト構成の理解**: ディレクトリ構成、主要モジュール、レイヤー構成を把握する
2. **アーキテクチャルール**: README, CLAUDE.md, ADR (Architecture Decision Records) 等のドキュメントから、プロジェクトのアーキテクチャルールを確認する
3. **既存パターンの調査**: 変更と同種の既存実装を検索し、確立されたパターンを把握する
4. **依存関係の分析**: import/require 文を分析し、依存方向が正しいか確認する
5. **変更の影響範囲の評価**: 変更が他のモジュールやレイヤーに与える影響を評価する
6. **整合性の判定**: チェックリストに基づき、既存アーキテクチャとの整合性を判定する

## よくある問題パターン

### レイヤー違反

- **症状**: UI コンポーネントが直接データベースクエリを実行、ビジネスロジックが HTTP リクエスト/レスポンスを直接操作
- **根拠**: レイヤードアーキテクチャの原則 — 各レイヤーは隣接レイヤーとのみ通信すべき
- **対処**: 適切なレイヤーにロジックを移動し、定義されたインターフェースを通じてアクセスする

### 循環依存

- **症状**: モジュール A が B に依存し、B が A に依存する（直接または間接）
- **根拠**: 循環依存はビルドエラー、テスト困難、変更の波及効果を引き起こす
- **対処**: 共通インターフェースの抽出、イベント駆動パターンの導入、またはモジュール構成の見直しを提案する

### 既存パターンとの不一致

- **症状**: プロジェクトで Repository パターンを使用しているのに、新しいコードが直接 DB アクセスを行う
- **根拠**: 一貫性はコードの予測可能性と保守性を向上させる
- **対処**: 既存パターンに合わせた実装に変更する。既存パターンに問題がある場合は、パターン自体の改善を別途提案する

### 過度な共有

- **症状**: 無関係なモジュール間でユーティリティや型定義を過剰に共有
- **根拠**: 過度な共有は見えない結合を生み、変更の影響範囲を予測困難にする
- **対処**: 共有の必要性を再評価し、必要に応じてモジュール内にコピーする（適度な重複は許容される）

### 責務の集中

- **症状**: 1つのモジュール/ファイルにプロジェクトの多くの機能が集中
- **根拠**: 責務の集中は変更のコンフリクト頻度を高め、理解を困難にする
- **対処**: 責務に基づいてモジュールを分割し、明確なインターフェースで接続する

## プロジェクトパターンの調査方法

アーキテクチャの整合性を判断するには、プロジェクトの既存パターンを調査する必要がある:

```bash
# ディレクトリ構成の把握
find . -type f -name "*.ts" | head -50

# 同種のファイルパターンの調査
find . -name "*Controller*" -o -name "*Service*" -o -name "*Repository*"

# import パターンの調査
grep -r "from.*import" --include="*.ts" | head -30

# 設定ファイルの確認
cat README.md CLAUDE.md .editorconfig tsconfig.json 2>/dev/null
```
