# パフォーマンス最適化設計ガイド

## 目的・スコープ

パフォーマンス改善、スケーリング対応の設計を行う際のガイド。
計測に基づくボトルネック特定と、効果的な最適化戦略の策定を含む。

**適用シチュエーション:**

- レスポンスタイム・スループットの改善
- データ量増加に伴うパフォーマンス劣化への対応
- スケールアップ / スケールアウトの設計
- フロントエンドパフォーマンスの最適化
- バッチ処理の高速化

## 設計チェックリスト

### 1. 計測と現状把握（最適化の前提）

- [ ] **計測なしの最適化は禁止** — ボトルネックを推測ではなく計測で特定したか
- [ ] パフォーマンス目標（SLO/SLI）を定義したか
- [ ] ベースラインの計測値を記録したか
- [ ] プロファイリングツールで実際のホットスポットを特定したか
- [ ] 現在のデータ量と将来の成長予測を把握したか

### 2. アルゴリズムとデータ構造

- [ ] ホットパスの時間計算量（Big-O）を分析したか
- [ ] データ量に対して適切なアルゴリズムを選択しているか
- [ ] 不要な計算の繰り返し（ループ内の再計算）を排除したか
- [ ] メモ化 / キャッシュで重複計算を削減したか
- [ ] ソート済みデータに対して線形探索ではなく二分探索を使用しているか
- [ ] 適切なデータ構造を選択しているか（HashMap vs List、Set vs Array）

### 3. データベースクエリ

- [ ] N+1 クエリ問題を排除したか
- [ ] EXPLAIN でクエリ実行計画を確認したか
- [ ] 適切なインデックスが設計されているか
- [ ] `SELECT *` ではなく必要なカラムのみ取得しているか
- [ ] 大量データの取得にページネーションを実装したか
- [ ] トランザクションの範囲が最小限か
- [ ] 適切な場所でクエリキャッシュを活用しているか

```sql
-- N+1 問題の例と修正
-- BAD: N+1（ユーザーごとに1クエリ）
SELECT * FROM users;
-- → 各userに対して: SELECT * FROM posts WHERE user_id = ?;

-- GOOD: JOIN または IN でまとめて取得
SELECT u.*, p.* FROM users u
LEFT JOIN posts p ON u.id = p.user_id;
```

### 4. I/O とネットワーク

- [ ] 同期 I/O がボトルネックになっていないか（非同期化を検討）
- [ ] 独立した I/O 操作を並列化しているか

```javascript
// BAD: 直列 I/O
const users = await fetchUsers();
const products = await fetchProducts();
const orders = await fetchOrders();

// GOOD: 並列 I/O
const [users, products, orders] = await Promise.all([
  fetchUsers(), fetchProducts(), fetchOrders()
]);
```

- [ ] API コールをバッチ化しているか（個別リクエストの最小化）
- [ ] タイムアウトを適切に設定しているか
- [ ] 接続プーリングを活用しているか

### 5. キャッシュ戦略

- [ ] キャッシュすべきデータを特定したか（読み取り頻度 >> 書き込み頻度）
- [ ] キャッシュレイヤーを適切に選択したか（メモリ、Redis、CDN）
- [ ] キャッシュの TTL（有効期限）を定義したか
- [ ] キャッシュ無効化（Invalidation）の戦略を設計したか
- [ ] キャッシュのヒット率を監視する仕組みを検討したか
- [ ] キャッシュ障害時のフォールバック（Cache Aside パターン）を設計したか

### 6. メモリ管理

- [ ] メモリリークの可能性がないか（イベントリスナー、クロージャ、循環参照）
- [ ] 大量データを一括でメモリに載せていないか（ストリーム処理を検討）
- [ ] 不要なオブジェクトの参照を適切に解放しているか
- [ ] バッファサイズが適切か

### 7. フロントエンド（該当する場合）

- [ ] Core Web Vitals（LCP、FID/INP、CLS）の目標値を定義したか
- [ ] 不要な再レンダリングを防止しているか（React: useMemo, useCallback）
- [ ] 大量リストに仮想スクロールを使用しているか
- [ ] 画像・アセットを最適化しているか（WebP、遅延ロード、適切なサイズ）
- [ ] バンドルサイズを最適化しているか（コード分割、Tree Shaking）
- [ ] レイアウトスラッシング（強制レイアウト再計算）を避けているか

## 設計手順

1. **目標設定**: パフォーマンス SLO を定義（レスポンスタイム、スループット、メモリ使用量）
2. **ベースライン計測**: 現在のパフォーマンスを計測・記録
3. **プロファイリング**: ホットスポットを特定（CPU、メモリ、I/O、ネットワーク）
4. **ボトルネック分析**: 最も効果の高い最適化ポイントを特定
5. **最適化設計**: 具体的な改善策を設計（アルゴリズム、キャッシュ、クエリ等）
6. **効果予測**: 各最適化の期待効果を見積もる
7. **計測計画**: 最適化後の計測方法とベンチマークを定義

## よくある失敗パターン

- **早すぎる最適化**: 計測前にパフォーマンスを推測して最適化 → 必ず計測してから最適化
- **計測なしの最適化**: 効果を検証せずに「速くなったはず」→ Before/After のベンチマーク必須
- **全部キャッシュ**: キャッシュ無効化の複雑さを無視 → キャッシュは必要な箇所にのみ適用
- **マイクロ最適化**: 全体の 0.1% の処理を最適化 → ホットパス（全体の 80% を占める処理）に集中
- **N+1 の見落とし**: ORM の便利さに隠れた大量クエリ → クエリログで実際の発行数を確認
