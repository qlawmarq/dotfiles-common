# API・インターフェース設計ガイド

## 目的・スコープ

REST API、GraphQL、gRPC、内部モジュールインターフェースの設計を行う際のガイド。
リソースモデリング、エンドポイント設計、エラーレスポンス、バージョニング戦略を含む。

**適用シチュエーション:**

- 新規 API エンドポイントの設計
- 既存 API の拡張・変更
- マイクロサービス間通信の設計
- 外部公開 API の設計
- GraphQL スキーマ設計
- gRPC サービス定義

## 設計チェックリスト

### 1. リソース設計

- [ ] リソースを名詞で表現しているか（`/users`、`/orders`、`/products`）
- [ ] リソースの階層関係が URL パスに反映されているか（`/users/{id}/orders`）
- [ ] コレクション（複数形）と個別リソース（単数 + ID）を区別しているか
- [ ] リソースの粒度が適切か（過度に細かい / 粗い リソースを避ける）
- [ ] サブリソースの深さが 3 階層以内か

### 2. HTTP メソッド・操作

- [ ] CRUD 操作に適切な HTTP メソッドを使用しているか
  - `GET`: 取得（安全・冪等）
  - `POST`: 作成
  - `PUT`: 全体更新（冪等）
  - `PATCH`: 部分更新
  - `DELETE`: 削除（冪等）
- [ ] 冪等性が必要な操作で冪等性を保証しているか
- [ ] GET リクエストに副作用がないか

### 3. リクエスト・レスポンス設計

- [ ] リクエスト/レスポンスの JSON スキーマを定義したか
- [ ] フィールド名の命名規約が統一されているか（camelCase / snake_case）
- [ ] 日付形式が ISO 8601（`2024-01-15T09:30:00Z`）か
- [ ] 列挙値が文字列定数で表現されているか（マジックナンバー禁止）
- [ ] レスポンスに不要な内部情報を含めていないか

### 4. ページネーション

- [ ] コレクション取得時にページネーションを実装しているか
- [ ] ページネーション方式を選択したか（オフセット / カーソル / キーセット）
- [ ] レスポンスに総件数・次ページリンクを含めているか
- [ ] デフォルトの件数制限と最大件数を定義したか

### 5. エラーレスポンス

- [ ] HTTP ステータスコードを正しく使用しているか
  - `400`: クライアントエラー（バリデーション失敗）
  - `401`: 未認証
  - `403`: 権限不足
  - `404`: リソース未存在
  - `409`: 競合（楽観的ロック失敗等）
  - `422`: 処理不能（ビジネスルール違反）
  - `429`: レート制限超過
  - `500`: サーバーエラー
- [ ] エラーレスポンスの形式が統一されているか
- [ ] エラーメッセージがクライアントに有用か（内部情報は含めない）
- [ ] エラーコード体系を定義したか（機械可読な識別子）

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Invalid input",
    "details": [
      { "field": "email", "message": "must be a valid email address" }
    ]
  }
}
```

### 6. 認証・認可

- [ ] 認証方式を選択したか（OAuth 2.0、JWT、API キー）
- [ ] トークンの有効期限・リフレッシュ戦略を定義したか
- [ ] エンドポイントごとの認可ルールを定義したか
- [ ] 認可エラー時に `401`（未認証）と `403`（権限不足）を正しく区別しているか

### 7. バージョニング

- [ ] バージョニング戦略を選択したか（URL パス `/v1/`、ヘッダー、クエリパラメータ）
- [ ] 後方互換性のポリシーを定義したか
- [ ] 破壊的変更の通知・移行計画を策定したか
- [ ] 廃止予定 API の Deprecation ヘッダーを設計したか

### 8. レート制限・キャッシュ

- [ ] レート制限を設計したか（エンドポイントごと / ユーザーごと）
- [ ] レート制限情報をレスポンスヘッダーに含めたか（`X-RateLimit-*`）
- [ ] キャッシュ戦略を設計したか（`Cache-Control`、`ETag`、`Last-Modified`）
- [ ] キャッシュ無効化の方針を定義したか

## 設計手順

1. **ユースケース整理**: API を利用するクライアントとユースケースを列挙
2. **リソースモデリング**: ドメインオブジェクトからリソースを抽出・定義
3. **エンドポイント設計**: 各リソースに対する操作とエンドポイントを定義
4. **スキーマ定義**: リクエスト/レスポンスの JSON スキーマを設計
5. **エラー設計**: エラーケースの列挙とレスポンス形式の統一
6. **横断的関心事**: 認証、レート制限、バージョニング、ページネーションの設計
7. **ドキュメント**: OpenAPI (Swagger) 仕様の作成

## よくある失敗パターン

- **RPC スタイルの REST**: `/createUser`、`/getOrders` → リソースベース `/users`、`/orders` にする
- **一貫性のない命名**: キャメルケースとスネークケースの混在 → プロジェクト全体で統一
- **バージョニングなしの破壊的変更**: クライアントが壊れる → 最初からバージョニングを導入
- **N+1 API コール**: 詳細取得のために個別リクエストが必要 → `include` / `expand` パラメータで関連リソースを埋め込み
- **過度な HATEOAS**: 不要な複雑性 → クライアントの実際のニーズに基づく
