# 外部サービス統合設計ガイド

## 目的・スコープ

サードパーティ API 連携、外部サービス統合、サービス間通信の設計を行う際のガイド。
抽象化レイヤーの設計、障害耐性、プロバイダー切り替え可能性を含む。

**適用シチュエーション:**

- 外部 API（決済、認証、メール、ストレージ等）の統合
- SaaS サービスとの連携設計
- マイクロサービス間の通信設計
- Webhook 受信の設計
- 外部データソースとの同期設計

## 設計チェックリスト

### 1. サービス評価

- [ ] API プロバイダーの公式ドキュメントを確認したか
- [ ] 利用規約、料金体系、使用制限を確認したか
- [ ] SLA（稼働率、レスポンスタイム）を確認したか
- [ ] レート制限（リクエスト/分、リクエスト/日）を把握したか
- [ ] サンドボックス / テスト環境が利用可能か確認したか
- [ ] SDK / クライアントライブラリの品質と保守状態を確認したか

### 2. 抽象化レイヤー設計

- [ ] ビジネスロジックを外部サービスの API に直接結合していないか
- [ ] 抽象インターフェースを定義し、外部サービスを実装詳細として分離したか
- [ ] プロバイダーの切り替えがインターフェース実装の差し替えで可能か
- [ ] テスト時にモック実装に差し替え可能か

```
# 推奨構造
interface PaymentGateway:
    charge(amount, currency, token) -> PaymentResult
    refund(payment_id, amount) -> RefundResult

class StripeGateway implements PaymentGateway:
    # Stripe固有の実装

class PayPalGateway implements PaymentGateway:
    # PayPal固有の実装

class MockGateway implements PaymentGateway:
    # テスト用モック
```

### 3. エラーハンドリング・耐障害性

- [ ] 外部サービスの障害時にシステム全体が停止しないか
- [ ] タイムアウトを適切に設定したか（接続タイムアウト、読み取りタイムアウト）
- [ ] リトライ戦略を設計したか（指数バックオフ + ジッター）

```
リトライ間隔 = min(base × 2^attempt + random_jitter, max_interval)
例: 1s → 2s → 4s → 8s（最大30s）
```

- [ ] サーキットブレーカーパターンを検討したか

```
状態遷移:
  Closed（正常）→ 連続エラー閾値超過 → Open（遮断）
  Open → 一定時間経過 → Half-Open（試行）
  Half-Open → 成功 → Closed / 失敗 → Open
```

- [ ] フォールバック動作を設計したか（キャッシュ応答、縮退機能、キュー蓄積）
- [ ] 冪等性を確保しているか（リトライ時の重複処理防止）

### 4. データ同期

- [ ] 同期方式を選択したか（リアルタイム / バッチ / イベント駆動）
- [ ] データの一貫性保証レベルを定義したか（強整合 / 結果整合）
- [ ] 同期の競合解決戦略を設計したか（Last Writer Wins / マージ / 手動解決）
- [ ] 同期の失敗時のリカバリ手順を設計したか
- [ ] データ形式の変換（マッピング）を明確にしたか

### 5. 認証・セキュリティ

- [ ] API 認証方式（OAuth 2.0、API キー、JWT）に対応した設計か
- [ ] 認証情報をシークレットマネージャーで管理しているか
- [ ] トークンの有効期限管理とリフレッシュを自動化しているか
- [ ] Webhook の署名検証を実装しているか
- [ ] 通信が HTTPS で暗号化されているか

### 6. 監視・運用

- [ ] 外部サービスの応答時間・エラー率を監視する仕組みを設計したか
- [ ] レート制限の消費状況を監視しているか
- [ ] 外部サービス障害時のアラートを設計したか
- [ ] API バージョン変更・廃止の追跡方法を定義したか
- [ ] コスト（API コール数）の監視を設計したか

## 設計手順

1. **要件整理**: 外部サービスに求める機能・非機能要件を明確化
2. **プロバイダー評価**: ドキュメント、SLA、制限事項、料金の確認
3. **抽象化設計**: インターフェース定義、プロバイダー実装の分離
4. **エラー設計**: タイムアウト、リトライ、サーキットブレーカー、フォールバックの設計
5. **データ設計**: データ同期方式、変換ルール、整合性保証の設計
6. **セキュリティ設計**: 認証、シークレット管理、通信暗号化
7. **監視設計**: メトリクス、アラート、コスト監視の設計

## よくある失敗パターン

- **直接結合**: ビジネスロジックが外部 API に直接依存 → 抽象化レイヤーで分離
- **フォールバックなし**: 外部障害でシステム全体が停止 → 縮退動作を設計
- **レート制限の無視**: 大量リクエストでブロックされる → レート制限を把握し、バッチ化・キューイングで対応
- **冪等性の欠如**: リトライで二重課金・二重送信 → 冪等キーの導入
- **SLA の不整合**: 自サービスの SLA が外部サービスの SLA より高い → 依存サービスの SLA を考慮した設計
