# バグ修正・エラー修正設計ガイド

## 目的・スコープ

バグの分析と修正計画の策定、障害対応の設計を行う際のガイド。
根本原因の特定から修正方針の決定、回帰テスト、再発防止までを含む。

**適用シチュエーション:**

- バグ報告に基づく修正計画の策定
- 本番障害の緊急対応設計
- 断続的に発生するバグの調査設計
- エラーログから発見された問題の修正計画

## 設計チェックリスト

### 1. 再現と事実確認

- [ ] バグを確実に再現できる手順を確立したか
- [ ] 再現環境（OS、ブラウザ、バージョン、データ状態）を特定したか
- [ ] 期待される動作と実際の動作の差分を明確にしたか
- [ ] 関連するログ・エラーメッセージ・スタックトレースを収集したか
- [ ] 再現できない場合、追加情報の収集計画を立てたか

### 2. 根本原因分析

- [ ] 5 Whys（なぜなぜ分析）で根本原因まで掘り下げたか
- [ ] 表面的な症状ではなく、根本原因を特定したか
- [ ] 影響を受けるコードパスを全て特定したか
- [ ] いつから発生しているか（リグレッション元のコミット）を調査したか
- [ ] 同じ根本原因による他のバグの有無を確認したか

**5 Whys の適用例:**

```
問題: ユーザーがログインできない
Why 1: セッショントークンが無効と判定される
Why 2: トークンの有効期限チェックがサーバー時刻に依存
Why 3: サーバー時刻がずれている
Why 4: NTP 同期が停止している
Why 5: NTP サービスの設定が更新時に上書きされた
→ 根本原因: デプロイ時の設定管理の不備
```

### 3. 影響範囲の評価

- [ ] 影響を受けるユーザー・機能の範囲を特定したか
- [ ] データの整合性に影響はあるか（データ修復が必要か）
- [ ] 他の機能やサービスへの波及を確認したか
- [ ] 重大度（Critical / High / Medium / Low）を判定したか

### 4. 修正方針の選定

- [ ] 複数の修正方法を検討したか
- [ ] 各方法のリスクと影響範囲を比較したか
- [ ] 根本的修正か暫定対応かを明確にしたか（暫定の場合、根本修正のチケットを作成）
- [ ] 最小限の変更で修正できる方法を優先しているか
- [ ] 修正が新たなバグを生まないことを確認する方法を定義したか

### 5. 修正の安全性

- [ ] 修正が既存の正常な動作を壊さないか
- [ ] ロールバック手順を準備したか
- [ ] 段階的リリース（カナリア / フィーチャーフラグ）が必要か検討したか
- [ ] 修正のデプロイ順序に依存関係はないか

### 6. 回帰テスト

- [ ] バグの再現テスト（修正前に失敗、修正後に成功）を設計したか
- [ ] エッジケースのテストを追加したか
- [ ] 関連する既存テストが全て通ることを確認する計画があるか
- [ ] 同じ根本原因の別パターンもテストでカバーしたか

### 7. 再発防止

- [ ] 同種のバグを防ぐ仕組み（バリデーション、型チェック、リンター）を検討したか
- [ ] コードの構造的な改善が必要か評価したか
- [ ] モニタリング・アラートの追加が必要か検討したか
- [ ] ドキュメント・コメントの更新が必要か確認したか

## 設計手順

1. **情報収集**: バグ報告、ログ、エラーメッセージ、再現手順を収集
2. **再現**: 確実な再現手順を確立（再現できるまで調査を続ける）
3. **原因特定**: コード調査、5 Whys、git blame / git bisect で根本原因を特定
4. **影響評価**: 影響範囲、重大度、データ影響を評価
5. **修正設計**: 複数の修正方針を検討し、最適な方法を選択
6. **テスト設計**: 回帰テスト、エッジケーステストを設計
7. **再発防止**: 構造的な改善、モニタリング強化を検討

**git を活用した調査:**

```bash
# リグレッション元の特定
git bisect start
git bisect bad HEAD
git bisect good <known-good-commit>

# ファイルの変更履歴
git log --oneline -20 -- <file>

# 特定コードの変更者
git blame <file>

# 特定コミットの変更内容
git show <commit>
```

## よくある失敗パターン

- **ショットガンデバッグ**: 原因不明のまま手当たり次第に修正 → 必ず根本原因を特定してから修正
- **症状の修正**: 根本原因ではなく表面的な症状だけを修正 → 5 Whys で掘り下げる
- **回帰テストなし**: 修正が正しいことの検証手段がない → 必ず再現テストを先に書く
- **影響範囲の見落とし**: 修正が別の箇所を壊す → 関連コードパスを全て確認
- **暫定対応の放置**: 「後で直す」が永遠に後回し → 暫定対応と同時に根本修正チケットを作成
