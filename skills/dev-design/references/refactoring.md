# リファクタリング設計ガイド

## 目的・スコープ

コード品質改善、技術的負債の解消のための計画を策定する際のガイド。
外部動作を変えずに内部構造を改善するための設計を行う。

**適用シチュエーション:**

- コードの可読性・保守性改善
- 技術的負債の計画的な解消
- 新機能追加の前準備としてのリファクタリング
- パフォーマンス改善のための構造変更
- テスタビリティ向上のための設計改善

## 設計チェックリスト

### 1. リファクタリング前提条件

- [ ] **テストカバレッジの確認**: 対象コードに十分なテストが存在するか
- [ ] テストがない場合、先にテストを追加する計画を立てたか
- [ ] リファクタリングの目標（何を改善するか）を明確に定義したか
- [ ] リファクタリングの範囲を限定したか（スコープクリープ防止）
- [ ] 機能変更とリファクタリングを別コミットで行う計画か

### 2. コードスメルの特定

以下のコードスメルのうち、該当するものを特定する:

**肥大化（Bloaters）:**
- [ ] 長すぎるメソッド（40 行超）
- [ ] 巨大なクラス（多すぎる責務）
- [ ] 長すぎるパラメータリスト（4 つ超）
- [ ] データの群れ（常にセットで使われる変数群）

**変更困難（Change Preventers）:**
- [ ] 変更の分散（1 つの変更が複数クラスに影響）
- [ ] 変更の集中（複数の理由で 1 つのクラスが変更される）
- [ ] 並行継承（1 つのサブクラス追加が別の継承階層にも必要）

**結合問題（Couplers）:**
- [ ] Feature Envy（他クラスのデータに過度にアクセス）
- [ ] 不適切な親密さ（クラス間の過度な依存）
- [ ] メッセージチェーン（`a.b().c().d()` のような連鎖）
- [ ] 仲介人（委譲ばかりのクラス）

**不要なもの（Dispensables）:**
- [ ] 死んだコード（使われていないコード）
- [ ] 重複コード
- [ ] 過剰な一般化（不要な抽象化）
- [ ] コメントで補っている難解なコード

### 3. リファクタリング手法の選択

| コードスメル | 推奨手法 |
|---|---|
| 長いメソッド | メソッド抽出（Extract Method） |
| 巨大なクラス | クラス抽出（Extract Class） |
| 長いパラメータリスト | パラメータオブジェクト導入 |
| 重複コード | メソッド / クラス抽出、テンプレートメソッド |
| Feature Envy | メソッド移動（Move Method） |
| 条件分岐の複雑化 | ポリモーフィズム / Strategy パターン |
| 不適切な親密さ | 移動 / インターフェース導入 |
| マジックナンバー | 定数抽出 |
| 型コードによる分岐 | サブクラス化 / State / Strategy |

### 4. 段階的実行計画

- [ ] 変更を小さなステップに分割したか
- [ ] 各ステップ後にテストが通ることを確認する計画か
- [ ] 各ステップが独立してコミット・レビュー可能か
- [ ] ステップ間の依存関係を整理したか
- [ ] 中断しても安全な中間状態を維持できるか

### 5. リスク評価

- [ ] 変更の影響範囲を特定したか（依存するコード・テストの洗い出し）
- [ ] 公開インターフェースの変更を最小限にしたか
- [ ] パフォーマンスへの影響を検討したか
- [ ] ロールバック可能な変更になっているか

## 設計手順

1. **現状分析**: 対象コードを読み、コードスメルを特定
2. **テスト確認**: 既存テストカバレッジの確認。不足していれば先にテスト追加
3. **目標設定**: リファクタリングの具体的なゴールを定義（「〜を改善する」ではなく「〜の状態にする」）
4. **手法選択**: コードスメルに適したリファクタリング手法を選択
5. **ステップ分割**: 小さく安全なステップに分割
6. **実行計画**: 各ステップの実行順序とテスト確認ポイントを定義

**Red-Green-Refactor サイクル:**

```
1. Red:   失敗するテストを書く（リファクタリング後の期待動作）
2. Green: テストが通る最小限の実装
3. Refactor: コード品質を改善（テストが通り続けることを確認）
```

## よくある失敗パターン

- **ビッグバンリファクタリング**: 大量の変更を一度に行う → 小さなステップに分割
- **テストなしのリファクタリング**: 動作保証なしで変更 → 必ずテストを先に確保
- **機能変更との混在**: リファクタリングと機能追加を同時に行う → 別コミットで分離
- **ゴールなきリファクタリング**: 「きれいにする」が目的化 → 具体的な改善目標を設定
- **過度な抽象化**: DRY 原則の過剰適用で不要な抽象レイヤーを追加 → Rule of Three（3 回目の重複で初めて抽象化）
