# 詳細設計ガイド

## 目的・スコープ

機能実装前のコンポーネント・モジュールレベルの設計を行う際のガイド。
クラス構造、メソッド設計、状態管理、エラーハンドリング戦略を具体化する。

**適用シチュエーション:**

- 新機能の実装設計
- モジュール・クラスの内部設計
- 状態管理やデータフローの設計
- コンポーネント間インターフェースの詳細化

## フェーズ前提条件（ゲートチェック）

**詳細設計に着手する前に、要件定義と基本設計の完了を確認すること。未達項目がある場合は該当フェーズに戻る。**

### 要件定義の完了確認（→ 未達なら requirements に戻る）

- [ ] **Why / Who / What** が明確に定義されているか
- [ ] 機能受入条件が具体的・測定可能な形で定義されているか
- [ ] 非機能要件（性能、可用性、セキュリティ）の基準が定義されているか

### 基本設計の完了確認（→ 未達なら architecture に戻る）

- [ ] アーキテクチャパターンが選択・合意されているか
- [ ] コンポーネント分割と責務が定義されているか
- [ ] コンポーネント間の通信方式・依存方向が決定しているか
- [ ] 技術スタックが選定されているか
- [ ] 横断的関心事（認証、ロギング、エラーハンドリング）の方針が決まっているか

> **判断基準**: 基本設計の決定事項が不明確なまま詳細設計を進めると、実装段階で根本的な手戻りが発生する。
> 不明な点があれば、先に上流フェーズの成果物を確定させること。

## 必須調査（設計前に必ず実施）

- [ ] 対象モジュールの既存コードを読み、現在の実装パターン・規約を把握したか
- [ ] 関連する既存テストを確認し、テスト戦略を理解したか
- [ ] 使用するライブラリ/フレームワークの公式ドキュメントで推奨パターンを確認したか
- [ ] `resolve-library-id` → `query-docs` で最新の API 仕様を取得したか

## 設計チェックリスト

### 1. 責務分割

- [ ] 各クラス・モジュールが単一の責務を持っているか（SRP）
- [ ] 責務の境界が明確で、他のモジュールと重複していないか
- [ ] 命名が責務を正確に反映しているか
- [ ] 既存のモジュール構成と一貫性があるか

### 2. 依存関係

- [ ] 依存の方向が上位レイヤー → 下位レイヤーの一方向か
- [ ] ドメインロジックが外部インフラに依存していないか（依存性逆転）
- [ ] 循環依存がないか
- [ ] 依存性注入（DI）を使い、テスト時にモック可能か
- [ ] 具象クラスではなく抽象（インターフェース）に依存しているか

### 3. インターフェース設計

- [ ] 公開 API（public メソッド・関数）が最小限か
- [ ] 引数と戻り値の型が明確か
- [ ] 副作用の有無が明示されているか（コマンド / クエリの分離）
- [ ] null / undefined / エラーの返し方が統一されているか
- [ ] 契約（事前条件・事後条件・不変条件）を定義しているか

### 4. 状態管理

- [ ] 状態を持つ箇所が最小限か（ステートレス優先）
- [ ] 状態遷移が明確に定義されているか（状態遷移図・状態マシン）
- [ ] 不正な状態遷移が防止されているか
- [ ] 並行アクセス時のスレッドセーフティを考慮しているか
- [ ] イミュータブルなデータ構造を優先しているか

### 5. エラーハンドリング戦略

- [ ] エラーの種類を分類したか（回復可能 / 回復不能、ビジネスエラー / システムエラー）
- [ ] エラーの伝播方式を決定したか（例外、Result 型、エラーコード）
- [ ] リトライ戦略（指数バックオフ等）を定義したか
- [ ] フォールバック・縮退動作を設計したか
- [ ] エラーメッセージがデバッグに十分な情報を含むか

### 6. データフロー

- [ ] データの入力から出力までのフローを図示・説明したか
- [ ] データ変換の各ステップが明確か
- [ ] バリデーションの実施ポイントを定義したか（境界での入力検証）
- [ ] データの一貫性をどこで保証するか決定したか

### 7. デザインパターンの適用

- [ ] 解決すべき問題にパターンが適合しているか（パターンのための設計ではない）
- [ ] プロジェクト内で既に使用されているパターンとの一貫性があるか
- [ ] パターン適用の理由を明記しているか

よく使われるパターンと適用場面:

| 問題 | 推奨パターン |
|---|---|
| オブジェクト生成の柔軟性 | Factory Method / Abstract Factory |
| 複雑なオブジェクト構築 | Builder |
| インターフェース変換 | Adapter |
| 複雑なサブシステムの簡素化 | Facade |
| アルゴリズムの切り替え | Strategy |
| 状態に応じた振る舞い変更 | State |
| 操作の取り消し・履歴 | Command / Memento |
| イベント通知 | Observer / Pub-Sub |
| コレクション走査 | Iterator |

## 設計手順

1. **ゲートチェック**: 要件定義・基本設計の完了を確認（未達なら該当フェーズに戻る）
2. **現行調査**: 対象コード・テスト・ライブラリの公式ドキュメントを確認
3. **構造設計**: モジュール分割、クラス構造、依存関係の定義
4. **インターフェース定義**: 公開 API の入出力・契約を設計
5. **内部設計**: 状態管理、データフロー、アルゴリズムの詳細化
6. **エラー設計**: エラーケースの列挙と対処方針の決定
7. **テスタビリティ検証**: 単体テスト・結合テストの書きやすさを確認

## よくある失敗パターン

- **God Class**: 1 クラスに多数の責務 → 責務ごとにクラスを分割
- **貧血ドメインモデル**: データだけ持ちロジックが外部 → ドメインオブジェクトにロジックを含める
- **Feature Envy**: 他クラスのデータに過度にアクセス → データとロジックを同じクラスに移動
- **早すぎる最適化**: 不要な複雑化 → 計測してからの最適化
- **設計パターンの乱用**: 単純な問題にパターンを適用 → 問題がパターンに適合する場合のみ使用
