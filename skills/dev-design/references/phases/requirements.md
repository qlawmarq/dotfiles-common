# 要件定義・要件分析ガイド

## 目的・スコープ

新機能の要件整理、ユーザーストーリー作成、受入条件定義、既存機能の変更要件分析を行う際のガイド。

**適用シチュエーション:**

- 新機能・新サービスの企画段階
- ユーザーからの機能要望の具体化
- 既存機能の改善・変更要件の整理
- プロジェクト開始時の要件定義

## 要件定義の原則

**要件定義では Why・What・Who を明確にする。How（実装方法）は扱わない。**

要件定義の目的は「ユーザーが何を求めているか」を正確に理解・定義することであり、
技術的な実現方法（アーキテクチャ、フレームワーク、データ構造等）は基本設計以降で決定する。

### 要件定義で答えるべき問い

| 問い | 内容 | 例 |
|---|---|---|
| **Why（なぜ）** | この機能/変更が必要な理由、解決すべき問題、ビジネスゴール | 「手動の集計作業に毎月20時間かかっており、自動化したい」 |
| **Who（誰が）** | 利用者、ステークホルダー、影響を受ける人・システム | 「経理チームの担当者が月次レポート作成時に使用する」 |
| **What（何を）** | 実現すべき機能・振る舞い、達成すべき成果 | 「売上データを自動集計し、月次レポートを生成する」 |
| **When（いつ）** | 時間的な制約、期限、頻度 | 「毎月第3営業日までに前月分を出力」 |
| **Where（どこで）** | 利用環境、動作環境の条件 | 「社内ネットワークからWebブラウザで利用」 |

> **How（どうやって）は要件定義に含めない。**
> 「React で実装する」「REST API で提供する」等の技術的決定は基本設計（architecture）で行う。
> 要件定義に How が混入すると、解決策の選択肢が不必要に制限される。

## 設計チェックリスト

### 1. Why・Who・What の明確化

- [ ] **Why**: この機能/変更がなぜ必要か、ビジネス上の動機・課題が明文化されているか
- [ ] **Who**: 誰がこの機能を使うか、利用者のペルソナ・役割が特定されているか
- [ ] **What**: 何を実現するか、期待される振る舞いが具体的に記述されているか
- [ ] How（技術的実現方法）が要件に混入していないか → 混入していれば分離する
- [ ] ユーザーの「本当に求めていること」を理解しているか（表面的な要望ではなく根本的なニーズ）

### 2. ステークホルダー分析

- [ ] 影響を受けるユーザー・システム・チームを特定したか
- [ ] 各ステークホルダーの期待と優先度を確認したか
- [ ] 決定権限者を明確にしたか

### 3. 機能要件

- [ ] ユーザーストーリー形式で記述したか（「〜として、〜したい。なぜなら〜」）
- [ ] 各機能の入力・処理・出力を明確にしたか
- [ ] 正常系・異常系のシナリオを網羅したか
- [ ] エッジケース（空入力、大量データ、同時操作）を考慮したか
- [ ] 既存機能への影響を分析したか

### 4. 非機能要件

- [ ] **性能**: レスポンスタイム、スループット、同時接続数の基準
- [ ] **可用性**: 稼働率、メンテナンスウィンドウ、障害時の復旧時間（RTO/RPO）
- [ ] **スケーラビリティ**: 想定データ量の成長率、ユーザー数の増加見込み
- [ ] **セキュリティ**: 認証・認可要件、データ保護レベル、コンプライアンス
- [ ] **運用性**: ログ・監視、デプロイ頻度、バックアップ

### 5. 制約条件

- [ ] 技術的制約（既存技術スタック、互換性要件）を明確にしたか
- [ ] ビジネス制約（予算、スケジュール、法規制）を明確にしたか
- [ ] 依存関係（他チーム、外部サービス、データソース）を特定したか

### 6. 受入条件と検証可能性

- [ ] 各要件に対して具体的・測定可能な受入条件を定義したか
- [ ] 受入条件が自動テストで検証可能か確認したか
- [ ] 「完了」の定義（Definition of Done）を明確にしたか

### 7. 優先度付け

- [ ] MoSCoW（Must/Should/Could/Won't）または類似手法で優先度を付けたか
- [ ] MVP（最小実行可能プロダクト）のスコープを定義したか
- [ ] 段階的リリース計画を検討したか

## 設計手順

1. **目的の明確化**: 何のために・誰のために作るのか、ビジネスゴールとの紐付け
2. **現状調査**: 既存システム・既存機能・関連ドキュメントの確認
3. **要件抽出**: ステークホルダーとの対話、ユーザーストーリー作成
4. **要件分析**: 一貫性・完全性・実現可能性の検証、矛盾の解消
5. **優先度付け**: ビジネス価値と技術的リスクの評価
6. **受入条件定義**: 各要件の検証基準を具体化
7. **レビュー・合意**: ステークホルダーとの最終確認

## よくある失敗パターン

- **ゴールドプレーティング**: 必要以上の機能を盛り込む → MVP を明確にする
- **曖昧な要件**: 「使いやすくする」→ 具体的な指標（レスポンス 200ms 以下等）に変換
- **非機能要件の欠落**: 機能だけ定義して性能・セキュリティを後回し → 最初から非機能要件チェックリストを適用
- **How の混入**: 「React で画面を作る」等の技術的決定を要件に含める → Why/What/Who に集中し、How は基本設計に委ねる
- **暗黙の前提**: 「当然こうだろう」という思い込み → 全ての前提を明文化
- **ステークホルダーの見落とし**: 運用チーム・セキュリティチームなど間接的関係者の漏れ
