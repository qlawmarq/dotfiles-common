# データモデル・DB設計ガイド

## 目的・スコープ

データベーススキーマ設計、データ構造設計、スキーマ変更の計画を行う際のガイド。
概念モデル・論理モデル・物理モデルの設計とマイグレーション戦略を含む。

**適用シチュエーション:**

- 新規データベーススキーマの設計
- 既存スキーマの変更・拡張
- データ構造（アプリケーション内）の設計
- NoSQL データモデリング
- マイグレーション計画の策定

## 設計チェックリスト

### 1. 概念モデル

- [ ] ビジネスエンティティとその関係を特定したか
- [ ] エンティティ間のカーディナリティ（1:1、1:N、N:N）を定義したか
- [ ] ユビキタス言語（ドメインの用語）を命名に使用しているか
- [ ] エンティティのライフサイクル（作成→更新→削除/アーカイブ）を考慮したか

### 2. 正規化・非正規化

- [ ] 第 3 正規形（3NF）以上に正規化したか（特別な理由がない限り）
- [ ] パフォーマンス要件に基づく戦略的な非正規化を検討したか
- [ ] 非正規化する場合、データ整合性の維持方法を設計したか
- [ ] 読み取り/書き込みの比率に基づいた設計判断をしたか

### 3. 整合性制約

- [ ] 主キー（PK）を適切に定義したか（自然キー vs サロゲートキー）
- [ ] 外部キー（FK）で参照整合性を保証しているか
- [ ] NOT NULL 制約を適切に設定したか
- [ ] UNIQUE 制約を必要な箇所に設定したか
- [ ] CHECK 制約でビジネスルールを表現しているか
- [ ] ON DELETE / ON UPDATE のカスケード動作を定義したか

### 4. インデックス戦略

- [ ] WHERE 句で頻繁に使用されるカラムにインデックスを設計したか
- [ ] JOIN に使用されるカラムにインデックスがあるか
- [ ] ORDER BY / GROUP BY で使用されるカラムを考慮したか
- [ ] 複合インデックスのカラム順序が最適か（選択性の高い順）
- [ ] インデックスの過剰設定による書き込み性能低下を考慮したか
- [ ] カバリングインデックスの活用を検討したか

### 5. 命名規約

- [ ] テーブル名: 複数形、スネークケース（`users`、`order_items`）
- [ ] カラム名: スネークケース、テーブル名プレフィックスなし
- [ ] 主キー: `id` または `{table}_id`
- [ ] 外部キー: `{参照テーブル単数形}_id`
- [ ] インデックス: `idx_{table}_{columns}`
- [ ] ブール値: `is_` / `has_` / `can_` プレフィックス
- [ ] タイムスタンプ: `created_at`、`updated_at`、`deleted_at`
- [ ] プロジェクト内の既存規約との一貫性を確認したか

### 6. データ保護

- [ ] 個人情報（PII）を特定し、適切な保護を設計したか
- [ ] 暗号化が必要なカラムを特定したか（保存時暗号化）
- [ ] ソフトデリート vs ハードデリートの方針を決定したか
- [ ] 監査ログ（誰が・いつ・何を変更したか）の設計をしたか
- [ ] データ保持期間（リテンションポリシー）を定義したか

### 7. マイグレーション計画

- [ ] マイグレーションスクリプトを作成したか
- [ ] ロールバック手順を定義したか
- [ ] 既存データの変換ロジックを設計したか
- [ ] ダウンタイムの有無と最小化策を検討したか
- [ ] 大量データのマイグレーション所要時間を見積もったか
- [ ] バックアップ・リストア手順を確認したか

### 8. NoSQL 特有の考慮事項（該当する場合）

- [ ] アクセスパターンに基づいたデータモデルか（クエリファースト設計）
- [ ] パーティションキー / ソートキーが適切か
- [ ] 非正規化によるデータ重複の更新戦略を設計したか
- [ ] 結果整合性の許容範囲を定義したか

## 設計手順

1. **要件確認**: データ要件、アクセスパターン、データ量の予測を確認
2. **概念モデル**: エンティティと関係の洗い出し（ER 図）
3. **論理モデル**: テーブル構造、カラム、制約の定義
4. **物理モデル**: インデックス、パーティション、データ型の最適化
5. **マイグレーション設計**: 既存データからの移行計画
6. **検証**: EXPLAIN での実行計画確認、パフォーマンステスト計画

## よくある失敗パターン

- **God テーブル**: 1 テーブルに大量のカラム → エンティティごとに分割
- **過度な正規化**: JOIN が多すぎてパフォーマンス低下 → アクセスパターンに基づく非正規化
- **インデックス不足**: 本番で遅いクエリが発覚 → アクセスパターンから逆算して設計
- **マイグレーション計画なし**: スキーマ変更がデプロイを阻害 → 最初からマイグレーション戦略を含める
- **EAV（Entity-Attribute-Value）の乱用**: 柔軟性のために型安全性を犠牲 → JSON カラムや適切なテーブル設計で代替
- **ソフトデリート非考慮**: 後から `deleted_at` を追加し全クエリに WHERE 条件が必要に → 最初から設計
