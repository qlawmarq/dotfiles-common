# セキュリティ設計ガイド

## 目的・スコープ

セキュリティ強化、脆弱性対応、認証・認可設計を行う際のガイド。
脅威モデリングに基づく防御設計と OWASP ベストプラクティスの適用を含む。

**適用シチュエーション:**

- 認証・認可システムの設計
- セキュリティ要件のある新機能の設計
- 脆弱性報告に基づくセキュリティ修正の設計
- API セキュリティの強化
- データ保護・暗号化の設計
- コンプライアンス要件への対応

## 設計チェックリスト

### 1. 脅威モデリング

- [ ] 保護すべき資産（データ、機能、インフラ）を特定したか
- [ ] 信頼境界（Trust Boundary）を定義したか
- [ ] 脅威アクター（誰が攻撃する可能性があるか）を想定したか
- [ ] STRIDE モデルで脅威を分類したか:
  - **S**poofing（なりすまし）
  - **T**ampering（改ざん）
  - **R**epudiation（否認）
  - **I**nformation Disclosure（情報漏洩）
  - **D**enial of Service（サービス拒否）
  - **E**levation of Privilege（権限昇格）
- [ ] 各脅威に対する対策を設計したか

### 2. 入力検証

- [ ] 全ての外部入力（ユーザー入力、API パラメータ、ファイルアップロード）を検証しているか
- [ ] ホワイトリスト方式（許可リスト）で検証しているか（ブラックリストではなく）
- [ ] SQL インジェクション対策: パラメータ化クエリ / プリペアドステートメントを使用しているか
- [ ] XSS 対策: 出力時のエスケープ / サニタイズを実装しているか
- [ ] コマンドインジェクション対策: シェルコマンドに外部入力を直接渡していないか
- [ ] パストラバーサル対策: ファイルパスに外部入力を含めていないか
- [ ] SSRF 対策: 外部から指定された URL にサーバーからアクセスしていないか

```python
# BAD: SQL インジェクション
cursor.execute(f"SELECT * FROM users WHERE id = {user_id}")

# GOOD: パラメータ化クエリ
cursor.execute("SELECT * FROM users WHERE id = %s", (user_id,))
```

### 3. 認証

- [ ] 認証方式を選択し、理由を明記したか（OAuth 2.0 / OIDC / SAML / JWT）
- [ ] パスワード保存に適切なハッシュを使用しているか（bcrypt / Argon2、**MD5/SHA-1 は禁止**）
- [ ] 多要素認証（MFA）の対応を検討したか
- [ ] セッション管理が安全か（十分なエントロピー、有効期限、HTTPOnly / Secure フラグ）
- [ ] ブルートフォース対策（アカウントロックアウト / レート制限）を実装しているか
- [ ] ログアウト時にセッションを確実に無効化しているか

### 4. 認可

- [ ] 認可モデルを選択したか（RBAC / ABAC / ACL）
- [ ] 全てのエンドポイントに認可チェックがあるか
- [ ] 水平権限昇格（他ユーザーのリソースへのアクセス）を防止しているか
- [ ] 垂直権限昇格（より高い権限の操作）を防止しているか
- [ ] 管理者権限の最小化を実践しているか
- [ ] 認可ロジックが一元化されているか（分散して不一致が生じていないか）

### 5. シークレット管理

- [ ] ソースコードにシークレットをハードコードしていないか
- [ ] ログ / エラーメッセージにシークレットが含まれていないか
- [ ] 環境変数またはシークレットマネージャーを使用しているか
- [ ] `.env` ファイルが `.gitignore` に含まれているか
- [ ] API キー / トークンのローテーション方針を定義したか

### 6. 暗号化

- [ ] 通信は TLS/HTTPS を使用しているか
- [ ] 保存データに暗号化が必要な箇所を特定したか（AES-256-GCM 推奨）
- [ ] 鍵管理の方針を定義したか（鍵のローテーション、アクセス制御）
- [ ] 古い / 脆弱な暗号アルゴリズムを使用していないか（MD5, SHA-1, DES, RC4 は禁止）
- [ ] PII（個人識別情報）の保護方針を定義したか

### 7. API セキュリティ

- [ ] レート制限を実装しているか
- [ ] CORS の設定が適切か（`*` ではなく特定オリジンを許可）
- [ ] セキュリティヘッダーを設定しているか（CSP, X-Frame-Options, X-Content-Type-Options）
- [ ] リクエストサイズの上限を設定しているか
- [ ] URL パラメータに機密情報を含めていないか（アクセスログに記録される）

### 8. 依存関係セキュリティ

- [ ] 既知の脆弱性（CVE）がないか確認したか（`npm audit` / `pip-audit` / `cargo audit`）
- [ ] 依存パッケージのバージョンをロックしているか
- [ ] 不要な依存パッケージを削除したか
- [ ] サプライチェーン攻撃のリスクを評価したか

## 設計手順

1. **資産特定**: 保護すべきデータ・機能・システムの洗い出し
2. **脅威モデリング**: STRIDE で脅威を分類・評価
3. **対策設計**: 各脅威に対する防御策を設計（多層防御）
4. **実装設計**: 入力検証、認証、認可、暗号化の具体的な設計
5. **依存関係確認**: 使用ライブラリの脆弱性チェック
6. **検証計画**: セキュリティテスト・ペネトレーションテストの計画

**セキュリティ設計の原則:**

- **最小権限**: 必要最小限の権限のみ付与
- **多層防御**: 単一の防御に依存しない
- **セキュアデフォルト**: デフォルトで安全な設定
- **ゼロトラスト**: 全てのアクセスを検証

## よくある失敗パターン

- **セキュリティの後付け**: 機能完成後にセキュリティを検討 → 設計段階から組み込む
- **Security by Obscurity**: 隠すことで安全と考える → 公開されても安全な設計にする
- **ハードコードされたシークレット**: ソースコードに API キーや DB パスワード → シークレットマネージャーを使用
- **クライアント側のみの検証**: フロントエンドの検証を信頼 → サーバー側でも必ず検証
- **過度な権限**: 「動くから」と管理者権限で実行 → 最小権限原則を適用
