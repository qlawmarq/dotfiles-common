# マイグレーション・移行設計ガイド

## 目的・スコープ

システム移行、フレームワーク更新、データベース移行、サービス移行の計画を策定する際のガイド。
リスクを最小化しながら安全に移行を完了するための設計を行う。

**適用シチュエーション:**

- フレームワーク・ライブラリのメジャーバージョンアップ
- データベースエンジンの移行（例: MySQL → PostgreSQL）
- モノリスからマイクロサービスへの分割
- クラウドプロバイダーの移行
- レガシーシステムのリプレース
- API バージョンの移行

## 設計チェックリスト

### 1. 移行戦略の選択

- [ ] 移行戦略を選択し、理由を明記したか

| 戦略 | 特徴 | 適用場面 |
|---|---|---|
| **Big Bang** | 一度に全体を切り替え | 小規模、強い整合性要件 |
| **Strangler Fig** | 段階的に新システムへ移行 | 大規模、リスク最小化 |
| **Parallel Run** | 新旧並行稼動で結果を比較 | 高信頼性要件、金融系 |
| **Blue-Green** | 2 環境を切り替え | ダウンタイムゼロ要件 |
| **Canary** | 少数ユーザーから段階的に展開 | ユーザー影響の段階的検証 |

### 2. 機能同等性（Feature Parity）

- [ ] 現行システムの全機能を洗い出したか
- [ ] 移行後に維持する機能と廃止する機能を分類したか
- [ ] 機能ごとの受入テストを定義したか
- [ ] 移行による機能変更・改善点を明確にしたか

### 3. データ移行

- [ ] 移行対象データの全量とデータ型を把握したか
- [ ] データ変換ルール（マッピング）を定義したか
- [ ] データの整合性検証方法を定義したか（件数照合、チェックサム、サンプル検証）
- [ ] 大量データの移行所要時間を見積もったか
- [ ] 増分データ（移行中に発生する新規データ）の取り扱いを設計したか

### 4. ロールバック計画

- [ ] ロールバック手順を定義したか
- [ ] ロールバックの判断基準（どの状態になったら戻すか）を決めたか
- [ ] ロールバック後のデータ整合性を確保する方法を設計したか
- [ ] ロールバックのリハーサルを計画したか
- [ ] ロールバック不可能な「ポイントオブノーリターン」を特定したか

### 5. 互換性・共存期間

- [ ] 新旧システムの共存期間を定義したか
- [ ] 共存期間中のデータ同期方法を設計したか
- [ ] 旧システムの廃止スケジュールを計画したか
- [ ] API の後方互換性を維持する方法を設計したか
- [ ] クライアント・依存サービスへの通知計画を策定したか

### 6. テスト計画

- [ ] 移行前テスト（リハーサル）を計画したか
- [ ] 本番相当のデータでの移行テストを計画したか
- [ ] 性能テスト（移行後のパフォーマンス検証）を計画したか
- [ ] ユーザー受入テスト（UAT）を計画したか
- [ ] 障害シナリオテスト（途中失敗時の復旧）を計画したか

### 7. リスク管理

- [ ] 移行に伴うリスクを洗い出したか
- [ ] 各リスクの発生確率と影響度を評価したか
- [ ] リスク軽減策を定義したか
- [ ] 移行実施のGo/No-Go判断基準を定義したか
- [ ] ダウンタイムの許容範囲とユーザー通知を計画したか

## 設計手順

1. **現状評価**: 現行システムの全体像、依存関係、データ量を把握
2. **目標定義**: 移行後のあるべき姿、成功基準を明確化
3. **戦略選択**: リスク許容度・規模に基づき移行戦略を選択
4. **計画策定**: マイルストーン、タイムライン、役割分担を定義
5. **データ移行設計**: データマッピング、変換ルール、検証方法を設計
6. **テスト設計**: リハーサル、性能テスト、UAT を計画
7. **ロールバック設計**: 復旧手順、判断基準を定義
8. **実行計画**: Go/No-Go 基準、実施手順、監視体制を確定

## よくある失敗パターン

- **ロールバック計画なし**: 問題発生時に戻せない → 必ずロールバック手順を先に設計
- **Big Bang 移行の過信**: 大規模システムで一括切り替え → Strangler Fig で段階的に移行
- **データ検証の不足**: 移行後にデータ不整合が発覚 → 複数の検証手法を組み合わせる
- **本番データでのテスト未実施**: テストデータでは発覚しない問題 → 本番相当データでリハーサル
- **共存期間の軽視**: 新旧システム間のデータ同期の複雑さを過小評価 → 共存期間の設計を入念に
