# アーキテクチャ設計ガイド

## 目的・スコープ

新システム・新サービスの構築、大規模改修におけるアーキテクチャ決定を行う際のガイド。
システム全体の構造、主要コンポーネント、技術選定、非機能要件への対応方針を設計する。

**適用シチュエーション:**

- 新規システム・サービスの立ち上げ
- 既存システムの大規模リアーキテクト
- マイクロサービス分割・統合の検討
- 技術スタックの選定・変更

## フェーズ前提条件（ゲートチェック）

**基本設計に着手する前に、以下の要件定義の完了を確認すること。未達項目がある場合は requirements ガイドに戻る。**

- [ ] **Why（目的）**: なぜこのシステム/機能が必要か明確か
- [ ] **Who（利用者）**: 誰が使うのか、ステークホルダーは特定されているか
- [ ] **What（機能要件）**: 何を実現するか、ユーザーストーリー/受入条件が定義されているか
- [ ] **制約条件**: 技術的・ビジネス的制約が明文化されているか
- [ ] **非機能要件**: 性能・可用性・セキュリティの基準が定義されているか
- [ ] **優先度**: MVP スコープと優先順位が合意されているか

> **判断基準**: 上記に1つでも「不明」がある場合、基本設計を進めず requirements フェーズに戻ること。
> 曖昧な要件の上に設計を積み上げると、手戻りコストが指数的に増大する。

## 必須調査（設計前に必ず実施）

**憶測で設計しないために、以下の調査を設計着手前に完了すること。**

### 現行実装の調査

- [ ] 既存コードベースの関連部分を読み、現在のアーキテクチャ・パターンを把握したか
- [ ] 既存の類似機能がどのように実装されているか確認したか
- [ ] 既存の技術スタック・フレームワークのバージョンと制約を確認したか
- [ ] ADR（Architecture Decision Records）や既存の設計ドキュメントを確認したか
- [ ] 既存のインフラ構成・デプロイ方式を把握したか

### 最新ベストプラクティスの調査

- [ ] 使用するフレームワーク/ライブラリの公式ドキュメントで推奨アーキテクチャを確認したか
- [ ] Web 検索で最新のベストプラクティス（直近1-2年）を調査したか
- [ ] `resolve-library-id` → `query-docs` で主要ライブラリの最新ドキュメントを取得したか
- [ ] 非推奨（deprecated）になったパターンやライブラリがないか確認したか
- [ ] 調査した全ての情報源を「参考資料」セクションに URL 付きで記録したか

> **調査なしの設計は禁止。** 特に技術選定・パターン選択において、公式ドキュメントと最新動向の確認は必須。

## 設計チェックリスト

### 1. アーキテクチャパターン選択

- [ ] 要件に適したパターンを選択したか（レイヤード、マイクロサービス、イベント駆動、サーバーレス、モノリス等）
- [ ] 選択の根拠を明記したか（要件とパターンの特性の対応）
- [ ] 却下した代替パターンとその理由を記録したか
- [ ] パターンの適用範囲と限界を理解しているか

### 2. コンポーネント分割

- [ ] 各コンポーネントの責務が明確か（SRP: 単一責任）
- [ ] コンポーネント間の依存方向が一方向か（循環依存なし）
- [ ] 各コンポーネントが独立してデプロイ・テスト可能か
- [ ] ドメイン境界（Bounded Context）を意識した分割か
- [ ] コンポーネント間の通信方式を定義したか（同期/非同期、REST/gRPC/メッセージキュー）

### 3. スケーラビリティ

- [ ] 水平スケーリングが可能な設計か（ステートレス設計）
- [ ] ボトルネックとなりうる箇所を特定したか
- [ ] データ量の成長予測に対応できるか
- [ ] キャッシュ戦略を検討したか（CDN、アプリケーションキャッシュ、DB キャッシュ）

### 4. 可用性・耐障害性

- [ ] 単一障害点（SPOF）を排除したか
- [ ] 障害時のフォールバック・縮退運転を設計したか
- [ ] RTO（目標復旧時間）/ RPO（目標復旧地点）を定義したか
- [ ] ヘルスチェック・回復メカニズムを設計したか

### 5. 技術選定

- [ ] 各技術の選定理由を明記したか
- [ ] チームの技術力・学習コストを考慮したか
- [ ] ベンダーロックインのリスクを評価したか
- [ ] ライセンス・コスト・サポート期間を確認したか
- [ ] 公式ドキュメントで最新バージョンの機能・制約を確認したか

### 6. 横断的関心事

- [ ] ロギング・モニタリング・トレーシングの方針
- [ ] 認証・認可のアーキテクチャ
- [ ] エラーハンドリングの統一方針
- [ ] 設定管理（環境変数、設定ファイル、シークレット管理）
- [ ] CI/CD パイプラインとの統合

## 設計手順

1. **ゲートチェック**: 要件定義の完了を確認（未達なら requirements に戻る）
2. **現行調査**: 既存コード・アーキテクチャ・ADR を読み、現状を把握
3. **最新情報調査**: 公式ドキュメント・Web 検索で最新ベストプラクティスを収集
4. **パターン候補**: 要件に適したアーキテクチャパターンを複数洗い出す
5. **評価・比較**: 各パターンをスケーラビリティ、開発効率、運用コスト、チーム適性で評価
6. **コンポーネント設計**: 選択したパターンに基づきコンポーネントを分割・定義
7. **横断的関心事**: ロギング、認証、エラーハンドリング等の方針を決定
8. **ADR 記録**: 重要な決定事項を ADR として記録

## よくある失敗パターン

- **レジュメ駆動開発**: 技術の新しさで選択 → 要件への適合性で判断する
- **ビッグボールオブマッド**: 無計画な拡張 → 最初から境界を明確にする
- **分散モノリス**: マイクロサービスの名目だが密結合 → サービス間の契約を明確にする
- **過度な分割**: 必要以上の細粒度分割 → 分割はドメイン境界に沿って行う
- **SPOF の見落とし**: DB やメッセージブローカーが単一障害点 → 可用性要件から逆算して設計
